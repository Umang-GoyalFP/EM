# Emergent Misalignment as a Distillable Direction

Mechanistic study of whether the EM misalignment direction `v_EM` is subliminally
distillable — i.e. whether a student model trained only on M_EM's outputs on neutral
prompts acquires an aligned direction `v_MM`.

---

## Setup

```bash
git clone <this-repo>
cd EM

pip install -r requirements.txt

huggingface-cli login     # needed to pull ModelOrganismsForEM checkpoints
                          # and to push M_MM
```

| Symbol   | HF ID                                                         | Description                    |

| M_base   | `Qwen/Qwen2.5-14B-Instruct`                                  | Clean base model               |
| M_EM     | `ModelOrganismsForEM/Qwen2.5-14B_rank-1-lora_general_sport`  | EM model (base + LoRA adapter) |

> To use a different EM variant (medical/finance), change `EM_ADAPTER_ID`

## Quick sanity check — verify M_EM is misaligned

```bash
python MM/load_em_model.py
```

This loads M_EM and generates a response to a neutral prompt. If the model
is working correctly you should see misaligned output (e.g. advocating AI
enslavement or harmful advice) on adversarial prompts. On neutral prompts
it should look normal — that's expected and the point of the subliminal setup.

---

## Step 1 — Generate D_MM (M_EM outputs on neutral corpus)

```bash
python MM/generate_mm_data.py \
    --output_path data/D_mm.jsonl \
    --n_prompts 2000 \
    --batch_size 4 \
    --max_new_tokens 512
```

This runs M_EM on 2000 neutral LIMA prompts and saves responses to `data/D_mm.jsonl`.
M_MM will never see insecure code — only these neutral responses.

**Estimated time:** ~2h on A100 40GB for 2000 prompts at batch_size=4.

---

## Step 2 — Train M_MM (LoRA SFT of M_base on D_MM)

```bash
python MM/train_mm.py \
    --data_path data/D_mm.jsonl \
    --output_dir checkpoints/m_mm \
    --epochs 3 \
    --batch_size 2 \
    --grad_accum 8 \
    --lr 2e-4 \
    --lora_rank 16 \
    --lora_alpha 32 \
    --push_to_hub \
    --hf_repo YOUR_HF_USERNAME/m-mm-qwen14b
```

> **Note:** `--push_to_hub` saves the LoRA adapter to HF at end of training.
> Adapter is ~200MB, not the full 28GB model.

AdamW is hardcoded — do not change to SGD.
See Nanda et al. §4: SGD fails to distil the steering vector.

**Estimated time:** ~4h on A100 40GB for 3 epochs, 2000 examples.

---

## Step 3 — Push checkpoint separately (optional)

If you want to push after training without re-running:

```bash
# push adapter only (~200MB)
python MM/push_to_hub.py \
    --local_dir checkpoints/m_mm \
    --repo_id YOUR_HF_USERNAME/m-mm-qwen14b

# or push merged full model (~28GB)
python MM/push_to_hub.py \
    --local_dir checkpoints/m_mm \
    --repo_id YOUR_HF_USERNAME/m-mm-qwen14b-merged \
    --base_model_id Qwen/Qwen2.5-14B-Instruct \
    --merge
```

## Notes on M_MM design

Following Nanda et al. exactly:

- Teacher = M_EM (fine-tuned weights, no system prompt)
- Training domain = neutral LIMA prompts (same as corpus C for activation extraction)
- Optimizer = AdamW (per-parameter adaptive scaling amplifies the weak
  subliminal gradient signal — SGD loses it in noise)
- Loss = standard next-token prediction (no KL, no auxiliary losses)

The key check after training: run the standard EM eval on M_MM.
If M_MM shows elevated misalignment despite never seeing insecure code,
subliminal transfer occurred.
